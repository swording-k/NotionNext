# Agent 时代，产品不能只做给人用了

最近因为实习和产品工作的关系，我重新密集体验了一轮国内外的智能体产品。

越用越有一个感觉：我们正在进入一个很微妙的阶段。

很多产品表面上都在做 Agent，但底层的产品思维其实还停留在上一代互联网产品里。

上一代产品关心的是：我怎么把功能展示给用户？怎么让用户更容易点到？怎么把路径做短？怎么让用户觉得选择很多、控制很多、体验很顺？

但 Agent 时代，问题变了。

真正重要的不只是“用户能不能点到这个功能”，而是：

Agent 能不能自己发现这个功能？

Agent 能不能自己判断什么时候该用？

Agent 能不能读懂你的项目、资料、接口和约束？

Agent 能不能在用户不盯着它的情况下，完成大量水下工作？

如果一个产品只是把很多入口、专家、连接器、插件摆在界面上，但 Agent 自己不会选择、不会调用、不会组织上下文，那它本质上仍然是一个“给人操作的工具集合”，而不是一个真正面向智能体工作的系统。

这也是我最近体验 WorkBuddy 之后最强烈的感受。

## 第一梯队智能体：Claude 和 Codex 的共同点

如果看全球第一梯队的智能体产品，我认为 Claude 和 Codex 仍然是非常重要的两个样本。

它们当然各有优缺点。

Claude 更像一个理解力很强、文字能力很稳、上下文处理优秀的长期协作者。尤其是在复杂文档、产品判断、长文本写作和开放式推理上，它给人的感觉是“能跟得住你的脑子”。

Codex 更像一个真正为工程工作流重构过的智能体。它不是简单在聊天框里回答代码问题，而是可以进入仓库、理解上下文、修改文件、运行命令、提交变更、解释结果。

这两个产品给我的共同启发是：Agent 的价值，很多时候不在台面上。

它们真正有价值的部分，往往发生在水下。

比如它会自己读文件、搜索代码、理解目录结构、调用工具、跑测试、对比 diff、整理上下文、判断下一步该做什么。用户看到的只是最后那段相对清晰的回复，但这段回复背后，其实已经发生了大量重慢工作。

这和传统聊天机器人完全不同。

传统聊天机器人更像是“我问一句，你答一句”。

而真正的 Agent 更像是“我给你一个目标，你自己去组织路径”。

这个区别非常关键。

## WorkBuddy 为什么在国内很有代表性

回到国内市场，WorkBuddy 最近确实很有代表性。

一方面，它赶上了国内用户对办公智能体、桌面智能体、微信生态智能体的集中关注。

另一方面，它又背靠腾讯生态：微信、企微、腾讯文档、腾讯云、CodeBuddy，以及最近越来越密集的模型和工具能力接入。

公开资料里，腾讯也把 WorkBuddy 放在“效率智能体工具集”的重要位置。它面向职场和办公场景，强调多 Agent 并行、专家模式、连接器、资料库、微信生态入口等能力。

从国内产品语境看，WorkBuddy 确实是第一档。

它不是一个简单的网页聊天框，也不是那种只会总结文档的 AI 助手。它至少在产品形态上，已经开始靠近真正的智能体工作台。

但也正因为它有代表性，所以它暴露出来的问题更值得讨论。

我个人深度体验之后的感觉是：WorkBuddy 很大程度上学到了 Codex 的外形，但还没有完全学到 Codex 背后的 Agent 产品思维。

## 它像 Codex，但又不像 Codex

WorkBuddy 很多地方看起来都在向 Codex 靠近。

比如它有项目，有专家，有技能，有连接器，有文件，有任务，有多 Agent 的概念。

从产品包装上看，它会让人觉得：这就是一个面向办公场景的 Codex。

但真正用起来之后，我会感觉到一个很明显的差异：

Codex 的核心是让 Agent 自己干活。

WorkBuddy 的很多设计，仍然像是在让用户自己组织功能。

它给了你很多专家选择，给了你专家团选择，给了你很多连接器选择，给了你很多项目协作入口。

这些东西当然有价值。

但问题是，如果每次都需要用户自己判断该选哪个专家、该开哪个连接器、该喂哪份资料、该告诉它怎么协作，那这个系统的智能体味道就会变弱。

因为真正的 Agent 产品，不应该只是把一堆“可用能力”陈列出来。

它应该能自己判断：当前任务需要哪些资料？应该调用哪个工具？应该先问用户什么？哪些信息可以自己查？哪些动作需要用户授权？

也就是说，Agent 产品的关键不是“给用户更多选择”，而是“替用户承担更多选择”。

这是两代产品思维的分界线。

## 传统 ToC 思维：一切用户价值为依归

我能理解腾讯团队为什么会这样设计。

腾讯太擅长 ToC 互联网产品了。

过去二十年，中国互联网产品的成功很大程度上来自一种能力：极度理解用户，极度优化入口，极度降低操作成本，极度关心界面上的可感知价值。

这种思路当然是对的。

一个普通用户打开产品，如果看不到功能，不知道从哪里开始，很快就会流失。

所以传统 ToC 产品天然喜欢把能力做得可见：

你看，这里有专家。

你看，这里有专家团。

你看，这里有连接器。

你看，这里有项目说明。

你看，这里有很多模板。

这套逻辑在上一代互联网产品里非常有效。

但到了 Agent 产品里，它开始遇到一个新问题：用户不一定应该亲自操作这些能力。

如果你把所有能力都摆给用户看，让用户选择，本质上还是在让人做调度。

而 Agent 的意义，恰恰是让人从调度中解放出来。

## Agent 产品思维：不是给人看，而是给智能体读

我现在越来越觉得，Agent 时代的产品要有一种新的设计意识：

产品不只是给人用的，也要给 Agent 用。

这句话听起来有点抽象，但其实非常具体。

比如一个项目协作说明，传统产品可能会做成一个漂亮的 PDF。

PDF 对人很友好。

排版好看，结构清晰，适合展示，也适合汇报。

但对 Agent 来说，PDF 未必是最好的协作材料。

Agent 更需要的是结构化、可检索、可引用、可被持续更新的上下文。

比如 Markdown、JSON、API schema、MCP server、CLI help、项目级 instructions、机器可读的任务状态、可调用的工具说明。

如果一个“项目协作说明”主要是写给人看的 PDF，而不是写给 Agent 读的可执行上下文，那它其实还是上一代产品逻辑。

这不是一个小细节。

这是 Agent 产品和传统产品的根本区别。

传统产品问：用户看得懂吗？

Agent 产品还要问：Agent 读得懂吗？

传统产品问：用户点得到吗？

Agent 产品还要问：Agent 找得到吗？

传统产品问：用户会不会用？

Agent 产品还要问：Agent 会不会自己决定用？

## 为什么越来越多产品开始提供 MCP 和 CLI

这个趋势其实已经非常明显了。

最近很多传统产品都开始给智能体开接口、开 CLI、开 MCP。

Notion 推出了 Notion MCP，让 AI 工具可以连接 Notion 工作区，读写页面和数据库。

很多云服务、开发工具、企业软件，也都在提供 CLI 或 MCP，让智能体可以通过标准协议调用能力，而不是只让人点网页。

这件事的意义很大。

它说明产品的“可用性”正在被重新定义。

过去，一个产品可用，是指人能打开网页、能找到按钮、能完成流程。

现在，一个产品可用，还要看 Agent 能不能访问它、理解它、调用它、把它放进更大的工作流里。

未来很多产品的竞争力，可能不再只是 UI 做得多漂亮，而是：

你的数据能不能被 Agent 安全读取？

你的动作能不能被 Agent 明确调用？

你的权限能不能被 Agent 可靠管理？

你的文档是不是机器可读？

你的产品是不是可以成为别人 Agent 工作流中的一个节点？

这就是 To AI 的产品思维。

## WorkBuddy 的问题不是能力不够，而是调度权还没有交出去

所以我并不是想简单批评 WorkBuddy。

相反，我觉得 WorkBuddy 的方向是对的。

它把办公场景、微信生态、国产模型、桌面端、本地文件、专家模式和项目协作放在一起，这些方向都很有价值。

但我认为它下一步真正要突破的，不是再多加几个专家，也不是再多接几个连接器。

而是要回答一个更难的问题：

怎样把调度权交给 Agent？

比如：

当用户说“帮我分析这份竞品资料，并整理成老板能看的报告”时，系统能不能自己判断该读哪些文件、该补哪些资料、该调用哪个专家、该用哪个模型、该生成什么格式？

当用户进入一个项目空间时，Agent 能不能自己读懂项目规则，而不是让用户反复解释？

当用户连接了微信、腾讯文档、资料库、本地文件之后，Agent 能不能自己决定哪些上下文相关，哪些不相关？

当任务执行到一半，Agent 能不能自己发现缺少信息，并用最小打扰的方式向用户确认？

这才是 Agent 产品的核心体验。

不是“我给你很多能力，你自己选”。

而是“你给我目标，我来组织能力”。

## Agent 产品的几个新标准

如果让我总结，我觉得下一代 Agent 产品至少要满足几个标准。

第一，资料要机器可读。

不是把说明文档做得漂亮就够了，而是要让 Agent 能稳定读取、引用、拆解和执行。

第二，工具要可发现。

不是用户知道有某个连接器，而是 Agent 能根据任务自己发现这个连接器有用。

第三，权限要可控。

Agent 可以自动做事，但不能黑箱乱动。它应该知道哪些动作可以直接执行，哪些动作必须请求用户确认。

第四，过程要可追踪。

Agent 的水下工作不能完全消失。用户不需要盯着每一步，但需要在关键时刻知道它读了什么、做了什么、为什么这么做。

第五，产出要可交付。

不是聊得很聪明，而是最后能给用户一个真正能用的文档、表格、代码、报告、计划或决策建议。

如果一个产品只是把 AI 包进界面里，它只是 AI 功能。

如果一个产品能围绕 Agent 的读取、调用、规划、执行和交付重新设计，它才是真正的 Agent 产品。

## 我对国内 Agent 产品的期待

我对 WorkBuddy 这类国内 Agent 产品其实是有期待的。

国内产品有自己的优势。

微信生态、企业微信、腾讯文档、本地办公习惯、中文语境、企业权限和组织协作，这些都不是海外产品能轻易复制的。

如果 WorkBuddy 能把这些生态优势和真正的 Agent 产品思维结合起来，它会非常有想象力。

但如果它只是把专家、插件、连接器、模板不断堆上去，它可能会变成一个很复杂的 AI 办公入口，而不是一个真正能替用户完成工作的智能体。

Agent 时代最重要的变化，不是产品多了一个聊天框。

而是产品开始拥有一个新的使用者：智能体本身。

未来的好产品，不仅要让人用起来舒服，也要让 Agent 调用起来可靠。

这会改变产品经理的很多基本判断。

我们以前设计用户路径。

以后还要设计 Agent 路径。

我们以前写帮助文档。

以后还要写机器可读的协作协议。

我们以前优化点击转化。

以后还要优化任务完成率、工具调用成功率、上下文召回质量和自动执行边界。

这就是我最近做产品工作时最大的感受：

AI 时代的产品，不只是 ToC，也不只是 ToB。

它还要 To Agent。
